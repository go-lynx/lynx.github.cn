# Database Plugin

The database plugin brings relational database connectivity into the Lynx runtime. Its job is not to dictate your ORM strategy. Its job is to handle **connection initialization, lifecycle ownership, and a stable injection entry** for application code.

In practice, the common path is: configure the database, let the plugin start, then inject the low-level driver or the higher-level client into the data layer.

## What it is for

- initializing and owning database connection pools
- tying database resources to application startup and shutdown
- exposing database capability through Wire or plugin getters
- letting you build `ent`, `gorm`, or your own repository layer on top

## Basic configuration

A common configuration shape looks like this:

```yaml
lynx:
  db:
    driver: mysql
    source: root:123@tcp(127.0.0.1:3306)/demo?parseTime=True
    min_conn: 10
    max_conn: 50
    max_idle_time: 30s
```

The important part is not the field names by themselves. The point is that database connection settings enter the runtime through `lynx.db`, and the plugin initializes the connection resource during startup.

## Application integration

If you use `ent`, a common pattern looks like this:

```go
import (
    "context"
    "database/sql"

    "github.com/go-kratos/kratos/v2/log"
    "github.com/go-lynx/lynx/plugin/db"
    "github.com/google/wire"
)

var ProviderSet = wire.NewSet(
    NewData,
    db.GetDriver,
)

type Data struct {
    db *ent.Client
}

func NewData(dri *sql.Driver, logger log.Logger) (*Data, error) {
    client := ent.NewClient(ent.Driver(dri), ent.Debug())

    if err := client.Schema.Create(context.Background()); err != nil {
        return nil, err
    }

    return &Data{db: client}, nil
}
```

Here `db.GetDriver` gives you the low-level connection entry. Whether you layer `ent`, `gorm`, or your own repository abstraction on top is still your application choice.

## Practical notes

- the plugin owns the connection resource, not your business ORM structure
- startup-critical connection settings belong in configuration, not scattered in code
- if you need multi-source, read/write split, or more advanced governance later, verify that the plugin capability matches that requirement
